## Background/Context

Orus M0 has several independent evidence producers, but they must agree on one
byte representation, one Unicode policy, one digest vocabulary, one support
decision, and one resource-failure convention. The
[implementation plan](../../IMPLEMENTATION_PLAN.md) therefore places this
serial contract task after the completed Nix/Bazel substrate and before the six
delivery lanes. No downstream lane may enter `DOING` until this task is
accepted.

[D-003](../../specs/02-DECISIONS.md) limits M0 to one pinned Linux x86-64
reference environment. [D-004](../../specs/02-DECISIONS.md) requires C++23,
Nix, and Bazel/Bzlmod. Resolved question `q-0012` fixes Glaze v7.5.0,
utf8proc, and OpenSSL 3 EVP behind an Orus-owned boundary; `q-0014` keeps
native and untrusted parsing plus canonical business logic in C++23 while
allowing pinned Python for cold tooling only after byte-identical parity is
proved.

This A3 applies [SPECS.md](../../SPECS.md), the
[Charter](../../specs/00-CHARTER.md), specs

## Problem Statement (the gap)

The completed M0-001 substrate pins the required dependencies and publishes a
bootstrap-checked reference document, but it deliberately does not implement
the reusable canonical core. At task selection there is no Orus-owned API that
can prove all of the following together:

- strict `M0-CANONICAL-JSON-v1` acceptance and emission independent of a
  third-party parser's permissive defaults;
- utf8proc NFC and OpenSSL EVP SHA-256 behavior hidden behind stable Orus
  types, typed errors, and bounds;
- byte-identical C++23/Python results for cold downstream tooling;
- authoritative build facts and truthful bounded reference-environment
  validation;
- non-interchangeable executable, package-tree, SBOM, and evidence-object
  identities;
- canonical shared schema/golden behavior for governance, performance, and
  corpus evidence; or
- exact-bound/first-over resource rejection, native parser fuzzing, and a
  package-scoped 70% coverage floor.

Without this boundary, downstream domains can each produce plausible JSON
that differs on duplicate names, Unicode, integer spelling, member order,
array semantics, digest subjects, or resource failure. Such drift would only
surface at the release join, after six lanes have built against incompatible
assumptions.

## Assumptions & Open Questions

### Assumptions

1. The accepted M0-001 artifacts named above are stable inherited evidence;
   this task never depends on an overwritable `.factory/results/.../result.json`.
2. Glaze may parse into private intermediate state, but Orus performs the
   byte-level prevalidation, NFC, unknown-field, numeric, relationship, and
   resource checks required by the specs before acceptance or identity.
3. Exact RSS/deadline boundary behavior can be tested deterministically through
   injectable accounting and monotonic-clock seams, provided production paths
   use real guards and the tests prove exact-bound acceptance and first-over
   rejection before allocation/identity side effects. These are contract tests,
   not a substituted external substrate.
4. The C++23 implementation is authoritative for native/untrusted parsing.
   Python consumes validated typed data or serves as an independent parity
   oracle; parity disagreement fails closed.
5. The package coverage tool may be extended backward-compatibly to enumerate
   native and Python packages, but it may not omit or weaken the already
   verified M0-001 package population.

### Open questions

None. `q-0012` and `q-0014` are answered, all referenced domain specs state
that they have no open question, and the completed predecessor supplies the
required tooling/pins.

## Root Cause Hypotheses / Key Risks (top 3-7)

| Risk | Hypothesis / failure mechanism | Observable harm |
|---|---|---|
| R1 — canonical divergence | C++, Python, or two domains accept different member order, duplicate names, Unicode, integer spelling, escaping, or terminal bytes. | Equal logical content receives different identities or noncanonical evidence is accepted. |
| R2 — dependency leakage | Glaze/utf8proc/OpenSSL types or permissive defaults escape into public contracts. | Downstream code becomes coupled to third-party ABI/behavior and can bypass Orus validation. |
| R3 — identity confusion | A generic digest, self-reference, subject substitution, or wrong metadata inclusion is accepted. | Executable, package, SBOM, or evidence bytes can be mislabeled while appearing hash-valid. |
| R4 — ownership creep | Shared schema work silently implements or redefines downstream producer/business workflows. | Parallel lanes lose a clear owner or later duplicate/undo M0-002 behavior. |
| R5 — off-by-one resources | Bounds are checked after allocation/identity, exact limits are rejected, or first-over inputs proceed. | Denial of service, partial authoritative output, or disagreement with the finite security inventory. |
| R6 — support/build overclaim | Empty/dirty build facts or a mismatch/unknown reference fact is presented as a clean supported result. | Misleading artifact provenance or a false platform compatibility claim. |
| R7 — test theater | Happy-path goldens pass while negative/fuzz/sanitizer execution is empty, coverage is aggregate-only, or logic is moved to an excluded path. | The contract freezes without evidence for malformed input or material code paths. |

## Check Plan

The executable and manual plan is [verification.md](verification.md). It has
five ordered runnable checks and two manual reviews. V1-V4 are the ledger's
exact dev, ASan, UBSan, and fuzz targets. V5 first creates fresh LCOV for the
retained build packages and the new contract packages, then invokes the
ledger's exact `//tools/coverage:package_gate -- --threshold=70` clause; this
prevents a stale report from satisfying the gate.

Negative semantics execute inside V1-V5, so exit code zero means the expected
malformed, relationship, resource, sanitizer/fuzz, and coverage-failure
fixtures were detected. V6 reviews the frozen contract/API/ownership surface.
V7 separates task-owned implementation from factory/audit metadata. No check
asserts process lifecycle state or reads `.factory/**` as product behavior.

## Andon Conditions

Stop implementation or verification and record the reason/evidence if any of
the following occurs:

- a required canonical rule, schema relationship, fixed digest, or literal
  error/bound cannot be implemented consistently with the approved specs;
- Glaze, utf8proc, or OpenSSL must be exposed through the public API, requires
  CMake, or cannot be used through the pinned M0-001 inputs;
- C++ and Python produce different bytes, digests, typed outcomes, or integer
  statistics for any shared golden;
- an exact-bound case fails, a first-over case allocates/accepts identity, or
  resource behavior depends on sleeping for the real multi-minute ceiling;
- a digest subject can be substituted, a self-reference is accepted, or a
  dirty/unknown/mismatched value becomes authoritative/supported;
- satisfying a cited criterion requires implementing a downstream producer,
  workflow, or shared surface not authorized by this task;
- a dependency pin/toolchain/acquisition change is required without explicit
  M0-001 owner coordination and reviewed evidence;
- any dev/ASan/UBSan/fuzz target is empty/skipped, fuzz executes zero inputs,
  or task-owned coverage is missing, below 70%, aggregate-only, or hidden by an
  unreviewed exclusion; or
- a product test asserts factory status/metadata, an implementation path falls
  outside scope, or any M1+ capability/support/security claim appears.
