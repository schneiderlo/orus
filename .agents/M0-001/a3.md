## Background/Context

Orus M0 requires one reproducible source-to-release path before any product
domain can build on it. [D-004](../../specs/02-DECISIONS.md) assigns the host
environment and packaging closure to Nix and every source compilation, link,
generation, test, benchmark, and release action to Bazel with Bzlmod. CMake is
prohibited. [D-003](../../specs/02-DECISIONS.md) limits M0 support to one pinned
Linux x86-64 reference environment, while
[D-011](../../specs/02-DECISIONS.md) preserves the later Linux x86-64 ELF/glibc
target boundary without claiming that later functionality exists.

The selected task is the first build-contract unit in the
[implementation plan](../../IMPLEMENTATION_PLAN.md). It has no predecessor and
owns the root Nix/Bzlmod/Bazel locks, configuration, toolchains, and dependency
ADR until it is `DONE`. Resolved questions `q-0003` and `q-0012` require the
author to select a current-stable mutually compatible Nixpkgs/Bazel/
Clang/LLVM/LLD/GCC set and to pin Glaze v7.5.0, utf8proc, OpenSSL 3 EVP,
GoogleTest, and Google Benchmark with exact immutable coordinates and digests.
Glaze's optional CMake path must not enter Orus.

This A3 applies [SPECS.md](../../SPECS.md), the
[Charter](../../specs/00-CHARTER.md), the
[build-environment spec](../../specs/10-build-environment.md), the accepted
foundation decisions, the [risk register](../../specs/03-RISKS.md), and the
[verifier-approved M0 review](../../specs/_reviews/2026-07-21_phaseM0_review.md).
No optional `technical-dive.md` or `plan.md` is needed: the owner boundaries,
contracts, sequence, risks, and executable checks fit in this A3 and
`verification.md` without duplicating the governing spec.

## Problem Statement (the gap)

At task selection, the repository contains specifications and the generated
implementation plan but no committed Nix flake/lock, Bazel/Bzlmod root/lock,
registered C++23 toolchains, configuration/test skeleton, or machine-run build
checks. Therefore a developer or verifier cannot yet prove that:

- one immutable input set determines the build;
- Nix and Bazel have non-overlapping ownership;
- Clang/LLD and GCC are selected rather than host tools;
- required configurations execute or fail with an explicit reason;
- dependency acquisition is bounded and separated from network-denied actions;
- the selected dependency stack is mutually compatible and CMake-free; or
- the one M0 reference-environment contract is exact and honestly scoped.

The gap is closed only by retained machine evidence, including negative
fixtures. Merely creating root files or naming configurations is insufficient.

  M0-001 selects and publishes the exact production `M0-REFENV-v1` document
  and validation ADR; M0-002 later implements and freezes the reusable bounded
  canonical-JSON/NFC/SHA-256 validator and its full resource matrix. M0-001
  must not duplicate that downstream core.
- Bootstrap dependency-admission records are part of M0-001 because pins may
  not enter locks unreviewed. M0-004 later implements their automated
  linter/reconciler and does not create a reverse dependency.
- The only factory substrate currently registered is a GitHub Actions
  substitute owned by later tasks. M0-001 uses the real local Nix/Bazel
  boundary; none of its checks is labeled end-to-end or passed through a
  substituted service.
- Canonical commands are non-interactive and run from the repository root.
  Product checks must not assert task statuses, ledger contents, or any
  `.factory/**` metadata.
- Factory coverage policy supplies the applicable baseline: each task-owned
  logic package exercised here must reach at least 70% line coverage. Vendor,
  generated, and fixture-only files may be excluded only through a reviewed
  finite manifest.

## Assumptions & Open Questions

### Assumptions

1. “Current stable” is evaluated by the Author when implementation starts.
   The selected exact coordinates are accepted only after mutual compatibility
   is exercised and recorded; the bundle does not invent them in advance.
2. Nix and Bazel can consume the required dependencies without invoking
   upstream CMake. If a candidate artifact cannot, the Author replaces or
   wraps it through an admitted CMake-free route rather than weakening D-004.
3. The canonical 70% baseline applies to task-owned logic packages used by
   build/audit tooling. Declarative Nix, Starlark, lock data, vendor code,
   generated code, and fixture-only data are not business-logic packages, but
   exclusions must still be finite and reviewed.
4. M0-001 publishes reference bytes and selection evidence; M0-002 remains the
   owner of the authoritative reusable canonicalization/validation code.

### Open questions

None. `q-0003` and `q-0012` are answered, the domain spec reports no open
questions, and no missing information prevents a safe bundle.

## Root Cause Hypotheses / Key Risks (top 3-7)

| Risk | Hypothesis / failure mechanism | Observable harm |
|---|---|---|
| R1 — split ownership | Nix or a wrapper performs source actions, or Bazel silently owns host/environment resolution. | Two drifting build graphs or a build that succeeds only on one host. |
| R2 — false pinning/fallback | Locks name versions, but toolchain registration or action lookup falls back to a host compiler/linker/header/library. | Unreproducible artifacts and misleading build facts. |
| R3 — incompatible or unsafe dependency stack | Independently selected Glaze/utf8proc/OpenSSL/test/benchmark/tool pins are ABI/closure incompatible, floating, unadmitted, or expose CMake. | Build failure, dependency bloat, hidden licenses, or a prohibited alternate path. |
| R4 — network/acquisition leakage | Fetching occurs during build/test actions, mutable or hash-mismatched bytes are promoted, or limits/credentials are not enforced. | Non-hermetic or supply-chain-ambiguous evidence. |
| R5 — configuration theater | Named sanitizer/fuzz/benchmark/GCC configurations silently skip, select the wrong toolchain, or have no executed negative fixture. | Green checks that exercised no intended behavior. |
| R6 — support and contract overclaim | A reference artifact omits exact predicates/coordinates or a passing non-reference host is presented as supported; M0-001 duplicates incomplete canonical-validator logic. | False compatibility claims or a boundary conflict with M0-002. |

## Check Plan

The executable and manual plan is
[verification.md](verification.md). It contains eleven ordered runnable checks
and two manual reviews. The runnable set includes the task ledger's exact
`nix flake check`, dev test, and GCC test commands, plus module-graph, release,
sanitizer, benchmark, hermeticity, prohibited-path, and 70% package-coverage
checks. All negative semantics execute inside those commands; exit code zero
means the expected failures were detected, not that a negative fixture was
skipped.

The plan exercises real local Nix/Bazel behavior and is therefore described as
build, test, integration, coverage, or audit evidence—not end-to-end evidence.
Factory/task lifecycle state and `.factory/**` contents are never assertions
inside product checks.

## Andon Conditions

Stop implementation or verification and record the reason/evidence if any of
the following occurs:

- a required input cannot be pinned by immutable coordinate and digest, or the
  selected set cannot pass the recorded compatibility matrix;
- Glaze or another dependency requires an Orus CMake invocation or a second
  source graph;
- any source action needs undeclared host input, credentials, or action-time
  network access;
- a named configuration silently skips, selects a host/wrong toolchain, or
  lacks an explicit applicability result;
- an acquisition first-over case promotes bytes, an invalid input reaches the
  read-only store, or an action executes after failed acquisition;
- task-owned logic coverage is below 70%, absent, aggregated in a way that
  masks a failing package, or relies on an unreviewed exclusion;
- the reference document cannot be expressed as exactly one narrow
  `validated_reference` contract without a broader platform claim;
- completing the work requires M0-002 canonical-core implementation, a later
  domain's owned surface, or an overlapping active task's root file; or
- any registered check is stale, non-interactive execution is impossible, or
  a verifier cannot distinguish task-owned changes from factory metadata.

  `.agents/M0-001/` as stable evidence. Factory run/check metadata remains a
  separate audit bucket, not implementation surface.
- Carry the package-scoped coverage ratchet forward: 70% at the contract
  branch, 75% at the integration join, and 80% at release exit, without moving
  logic into excluded paths.
